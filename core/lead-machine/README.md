# Lead Machine

Single-command runner for the Webconsult lead intelligence flow.

It coordinates existing modules without becoming a new scoring engine:

```text
fast discovery -> lead-pack-runner -> selected-lead enrichment when requested
```

It does not implement scoring, outreach, frontend, saved searches, or sales scripts. Balanced discovery is the default and combines Brreg official identity with Google Places public presence when both are available. Optional Proff enrichment belongs to selected-lead Deep enrichment and is gated behind confirmed org.nr.

## Usage

```bash
cd core/lead-machine
npm run lead-machine -- \
  --query "advokater i Gol" \
  --provider balanced \
  --max-results 5 \
  --search-scope strict \
  --enrich-company-profile false
```

## Discovery Providers

- `balanced`: default; Brreg-first official identity plus Google Places presence enrichment.
- `brreg`: official identity discovery by NACE and municipality.
- `google-places`: public presence discovery only.
- `brave`: web fallback when configured.

## Search Scope

- `strict`: exact-location only, default
- `nearby`: reserved for future radius behavior
- `regional`: explicit fallback; non-exact candidates are marked `regional_fallback`

## Outputs

```text
core/lead-machine/runs/<run-id>/
  discovery/
    lead-candidates.json
    discovery-summary.json
    handoff.jsonl
  lead-packs/
    lead-packs.json
    lead-packs.csv
    summary.json
  lead-machine-summary.json
```

The terminal output and `lead-machine-summary.json` are meant to be operator-readable. They include:

- query
- provider
- searchScope
- maxResults
- outputDir
- sourceRunPath
- leadPackOutputPath
- totalDiscovered
- includedLeadCount
- totalExcludedByLocation
- locationQualityCounts
- callPriorityCounts
- lowSupply
- fallbackAvailable
- fallbackUsed
- recommendedExpansion
- companyProfileEnabled
- companyProfileCounts
- economyStatus
- nextRecommendedAction


## Interpreting Results

- `strict` with `includedLeadCount=0` and `fallbackAvailable=true`: run again with `--search-scope nearby` or `regional`.
- `strict` with `lowSupply=true`: exact-location supply is limited; review exact leads first or expand if volume matters.
- `regional` with `fallbackUsed=true`: fallback leads are not local leads. Review location warnings before using them.
- Results with HIGH leads: review HIGH first.
- Results with no HIGH but MEDIUM leads: use MEDIUM as the shortlist layer.

## Product Boundary

The machine owns discovery, enrichment, ranking, evidence, caution, and export.

The seller owns angle, wording, outreach, timing, relationship, and closing.

No prepared sales scripts are generated by default.

## Run Modes

`--mode fast` creates lead packs directly from discovery output and skips full website audit. It is intended for quick seller scans: Brreg/company identity, Google Places presence, location quality, basic contact context, and visible warnings.

Fast and selected-lead Deep use `core/company-profile` with the local file cache enabled for Brreg/company identity lookups. The cache lives in `.cache/company-profile/`, keeps successful non-error profiles for 30 days by default, and avoids turning temporary Brreg failures into permanent results.

`--mode deep` should be understood as selected-lead enrichment. In the CLI it can still run the richer pipeline when requested, but product UX should enrich one selected lead rather than rerun the entire list by default. Website audit is one Deep module, alongside Brreg verification, contactability refresh, evidence/caution updates, and later Proff/social/recent activity modules.

Example:

```bash
npm run lead-machine -- --query "advokater i Moss" --provider balanced --max-results 20 --search-scope strict --mode fast
```


## Selected-Lead Enrichment Contract

Deep enrichment should:

- run on one selected lead when triggered from the desk
- preserve the rest of the Fast scan list unchanged
- append deeper evidence and caution instead of hiding uncertainty
- treat website audit as one module, not the whole product
- try to find email from the audited website/contact pages when a website exists
- keep `organizationNumber` confirmed-only and expose candidate/manual-verify states
- keep Proff/economy optional and run it only for confirmed org.nr when `PROFF_API_KEY` is configured
- avoid scripts, call openers, and outreach automation

Planned module stack:

- Brreg/company identity verification
- website audit
- contactability refresh
- seller leverage summary
- Proff/economy after confirmed org.nr
- social/source signals
- decision-maker hints
- recent activity and company fit
